Audit complet — Page Messages
Architecture globale
Trois couches : liste des canaux (messages.tsx) → écran de conversation (channel/[id].tsx) → composants (MessageBubble, MembersModal, modals). L'état vient de AppContext (reducer + refs + realtime Supabase). Globalement le code est bien structuré, mais voici tous les problèmes identifiés.

🔴 Bugs bloquants / comportements incorrects
1. Regex @\w+ incompatible avec les noms à espaces (channel/[id].tsx ligne 270)
/@(\w*)$/ ne capture que les caractères alphanumériques. Les noms comme "Jean Dupont" ne peuvent pas être mentionnés — l'autocomplete s'arrête à @Jean. L'insertion via insertMention injecte le nom complet (avec espace), mais le trigger ne le détecte pas. Résultat : l'autocomplete ne se déclenche jamais pour les noms composés.

2. readBy jamais mis à jour dans Supabase
Le badge "Vu par N" est affiché mais readBy dans la base reste toujours []. MARK_CHANNEL_READ_BY met à jour l'état local uniquement, sans jamais écrire dans Supabase. Le compteur est donc toujours à 0 pour tous les utilisateurs après rechargement.

3. Calcul des non-lus : comparaison de timestamp potentiellement fausse (AppContext ligne 1924-1926)
frTimestampToISO(m.timestamp) > lastRead compare une chaîne ISO avec '0' quand lastReadByChannel[ch.id] n'est pas défini. La comparaison '2026-04-05...' > '0' fonctionne par chance (lexicographique), mais c'est un comportement non garanti et fragile.

4. Profils non filtrés par organisation (AppContext ligne 1256)

supabase.from('profiles').select('id, name, role, role_label, email')
Récupère tous les profils de toutes les organisations. Conséquence : dans le modal "Ajouter membre" et dans l'autocomplete @mention, des utilisateurs d'autres orgs apparaissent. Fuite de données multi-tenant côté client.

5. Canaux DM non persistés dans Supabase
Les DMs n'ont pas de ligne dans la table channels. Ils sont reconstruits depuis les channel_id des messages ou depuis pendingDmChannelIds (AsyncStorage uniquement). Si l'utilisateur change d'appareil ou vide le cache, les conversations DM "en attente" (sans message échangé) disparaissent.

🟠 Problèmes de données / synchronisation
6. pinnedChannelIds stocké uniquement en AsyncStorage
Les canaux épinglés ne sont pas synchronisés sur Supabase. Un changement d'appareil remet tout à zéro. Idem pour lastReadByChannel — les badges de non-lus sont toujours à zéro sur un nouvel appareil.

7. Chargement limité à 500 messages sans pagination
supabase.from('messages').select('*').limit(500) charge les 500 messages les plus récents. Les plus anciens sont inaccessibles : pas de "charger plus", pas d'offset. La recherche globale ne voit elle aussi que ces 500 messages.

8. channelMembersOverride stocké uniquement en AsyncStorage
L'override de membres est local à l'appareil. Un admin qui ajoute un membre sur son téléphone ne le verra pas sur sa tablette (sauf si le canal est dans la table channels et que le realtime propage la mise à jour).

🟡 Problèmes de logique / UX
9. Action "Créer une réserve" affichée sur tous les types de messages
Elle apparaît même sur les messages notification, system, ou les messages vides (sans contenu textuel, seulement une photo ou un élément lié). Elle devrait être conditionnée à msg.type === 'message' && msg.content.

10. Suppression de message uniquement par l'expéditeur, sans rôle admin
selectedMsg?.isMe est la seule condition. Un admin ou conducteur ne peut pas supprimer les messages des autres, même en cas de contenu inapproprié.

11. Header du canal : compteur de membres manquant pour les canaux entreprise

(isDMChannel || isGroupChannel) && liveMembers.length > 0
  ? '…membres'
  : '…messages'
Les canaux entreprise n'affichent pas de membres (liste vide sur les canaux virtuels), donc le compteur affiche "N messages" au lieu de quelque chose de plus utile.

12. KeyboardAvoidingView sur Android utilise behavior='height'
Sur Android, height peut provoquer des comportements erratiques (le clavier fait sauter le layout). La pratique recommandée est padding sur les deux plateformes, ou aucun comportement sur Android.

13. renderItem dans le FlatList : closures potentiellement stale
Le useCallback à la ligne 433 liste [color, user?.name, channelId, sitePlans] comme dépendances, mais capture également router, reserves, visites, oprs via handleLinkedItemPress et handleNotifPress. Ces références peuvent être stale si elles changent entre les renders.

🔵 Problèmes de performance
14. lastMessageByChannel : O(N×M) sans indexation

messages.filter(m => m.channelId === ch.id)  // pour chaque canal
Avec 500 messages et 20 canaux = 10 000 itérations à chaque render. Il faudrait grouper les messages par channelId en une seule passe.

15. unreadByChannel dans AppContext sans useMemo
Recalculé à chaque re-render de n'importe quel consommateur du contexte, incluant frTimestampToISO pour chaque message de chaque canal.

16. FlatList de 500 messages sans pagination locale
Tous les messages sont dans le listItems array et rendus par le FlatList. Sur des canaux très actifs, ça peut peser lourd. Pas de fenêtrage sur les 100-200 derniers messages avec chargement à la demande.

🟢 Points bien faits
INCOMING_MESSAGE déduplique par ID (pas de doublon realtime) ✅
isMe calculé depuis le nom utilisateur en mémoire, pas depuis is_me DB (résistant au spoofing) ✅
Typing indicator avec cleanup propre des timeouts ✅
liveChannelName et liveMembers réactifs (mis à jour si le canal change pendant que l'utilisateur est dans le canal) ✅
Le canal entreprise est maintenant read-only pour la gestion des membres ✅
Les dates séparatrices dans la liste sont correctement gérées ✅
La recherche dans le canal est locale (instantanée) ✅
Les réactions inline fonctionnent correctement ✅
La protection super_admin sur la page messages est en place ✅
