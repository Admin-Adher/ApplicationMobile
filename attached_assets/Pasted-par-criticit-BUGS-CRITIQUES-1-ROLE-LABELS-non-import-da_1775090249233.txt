🔴 BUGS CRITIQUES
1. ROLE_LABELS non importé dans AppContext.tsx — ReferenceError potentiel

AppContext.tsx ligne 1074 utilise ROLE_LABELS[p.role as UserRole] dans loadAll() (mode Supabase), mais cette constante est définie uniquement dans AuthContext.tsx et n'est jamais importée dans AppContext. Si un profil Supabase n'a pas de role_label, cela provoque un ReferenceError à l'exécution.

2. Comparaison de timestamps en format français — Messages non lus incorrects

Les timestamps de messages sont générés avec nowTimestampFR() (format dd/mm/yyyy HH:mm). La comparaison des non-lus utilise l'opérateur > sur des strings :

m.timestamp > lastRead  // comparaison lexicographique !
Avec le format français, "02/04/2026" est inférieur à "31/03/2026" en tri alphabétique car "0" < "3". Le compteur de messages non lus peut donc être complètement faux.

3. File d'opérations hors-ligne jamais rejouée

NetworkContext accumule des opérations dans une queue (enqueueOperation) quand l'appareil est hors-ligne. Lors de la reconnexion, le code affiche seulement une alerte "vérifiez et ressaisissez". La queue n'est jamais traitée automatiquement — c'est une fausse promesse de offline-first.

🟠 BUGS MAJEURS
4. updateTask ne persiste pas chantierId vers Supabase

La fonction updateTask dans AppContext (lignes 1891–1895) envoie une mise à jour Supabase sans inclure chantier_id. Si l'on déplace une tâche entre chantiers, la modification n'est pas enregistrée en base.

5. stateRef.current obsolète après dispatch — Race condition systémique

stateRef est mis à jour dans un useEffect qui s'exécute après le rendu. Des fonctions comme updateVisite, updateLot, updateOpr, deleteVisite, deleteLot, deleteOpr appellent persistMockXxx(stateRef.current.xxx.map(...)) immédiatement après dispatch. La persistence locale lit donc un état obsolète (celui d'avant la modification).

6. Photos orphelines si la création de réserve échoue sur Supabase

Dans reserve/new.tsx, les photos sont ajoutées via addPhoto() APRÈS addReserve(). Si la réserve est rejetée par Supabase (rollback du dispatch DELETE_RESERVE), les photos restent dans le context photos et persistent localement — photos orphelines, jamais nettoyées.

7. Données mock persistées même avec Supabase configuré (tables vides)

Dans loadAll() (lignes 1085–1098), si Supabase est configuré mais que la table chantiers est vide, le code charge les MOCK_CHANTIERS et les persiste dans AsyncStorage avec persistMockChantiers. Lors du premier addChantier, un doublon est créé : une fois en Supabase, une fois en local. Au prochain chargement, les deux sources sont fusionnées.

8. addSitePlanVersion ne synchronise pas avec Supabase

La fonction crée une nouvelle version de plan et met à jour le plan parent localement, mais n'insère jamais en Supabase. Toutes les nouvelles versions de plans sont perdues après reconnexion ou réinstallation.

9. migrateReservesToPlan ne synchronise pas avec Supabase

Similaire au point 8 — la migration de réserves d'un plan à l'autre n'est jamais envoyée à Supabase.

🟡 BUGS MOYENS
10. Incohérence de couleur pour la priorité "Critique" (3 valeurs différentes)

Endroit	Couleur critique
constants/colors.ts (C.critical)	#DC2626 (rouge)
reserve/[id].tsx PRIORITY_COLORS (UI)	#EF4444 (rouge différent)
reserve/[id].tsx priorityColors (PDF)	#7C3AED (violet !)
Le PDF génère les réserves critiques en violet, l'UI les affiche en rouge.

11. persistMockMessages déclenché à chaque changement de messages

useEffect(() => {
  if (!isSupabaseConfigured) {
    persistMockMessages(state.messages);  // écrit dans AsyncStorage à CHAQUE changement
  }
}, [state.messages]);
Toute réaction, lecture ou mise à jour de message déclenche une réécriture complète de tous les messages en AsyncStorage. Impact performance significatif à mesure que le volume de messages grandit.

12. Double notification pour une réserve critique ET en retard

Si une réserve est à la fois critical et overdue, NotificationsContext génère deux notifications distinctes (crit_${id} et late_${id}). L'utilisateur voit la même réserve deux fois dans la liste des alertes.

13. Comment.createdAt en format ISO (YYYY-MM-DD) vs reste en (dd/mm/yyyy)

Les commentaires et addTaskComment créent :

createdAt: new Date().toISOString().slice(0, 10)  // "2026-04-01"
Alors que les réserves, tâches et visites stockent les dates en dd/mm/yyyy. La fonction formatDate gère les deux mais l'incohérence peut causer des bugs de tri ou d'affichage à l'avenir.

14. AuthGuard — segments[0] peut être undefined

const inPublic = PUBLIC_SEGMENTS.includes(segments[0] as string);
Si segments est vide (cas de démarrage), segments[0] est undefined. Le cast as string masque le problème mais PUBLIC_SEGMENTS.includes(undefined as string) retourne false, ce qui peut rediriger incorrectement vers /login lors de l'initialisation.

15. PointageContext et ReglementaireContext sans synchronisation Supabase

Le pointage individuel (TimeEntry) et les documents réglementaires (RegulatoryDoc) ne sont jamais synchronisés avec Supabase. Ils sont perdus si l'utilisateur change d'appareil ou efface son cache.

16. reloadApp() inopérant sur mobile

La fonction d'erreur dans _layout.tsx appelle window.location.reload() uniquement pour le web. Sur iOS/Android, le bouton "Redémarrer" n'a aucun effet — l'app reste bloquée sur l'écran d'erreur.

🔵 INCOHÉRENCES & CODE MORT
17. components/ErrorBoundary.tsx est du code mort

La classe ErrorBoundary dans components/ErrorBoundary.tsx n'est importée nulle part dans l'app. app/_layout.tsx définit sa propre ErrorBoundary en fonction composant. Le fichier components/ErrorBoundary.tsx est inutile.

18. Lot.number accédé via as any cast dans reserve/new.tsx

(lot as any).number  // ligne 448
Le champ number est pourtant défini dans l'interface Lot (number?: string). Le cast as any est inutile et masque un accès qui pourrait être typé correctement.

19. seedDemoUsers peut interférer avec une connexion utilisateur en cours

La fonction seedDemoUsers se connecte temporairement comme chaque utilisateur démo pour créer les profils. Si un vrai utilisateur essaie de se connecter pendant cette fenêtre (~30s), la déconnexion automatique (signOut) peut perturber sa session.

20. MOCK_TODAY via toLocaleDateString avec locale forcée

const MOCK_TODAY = new Date().toLocaleDateString('fr-FR', {...})
Bien que la locale soit forcée, sur certains navigateurs ou versions de Node en environnement web SSR, le format exact peut varier. Il serait plus sûr d'utiliser formatDateTimeFR(new Date()) déjà disponible dans lib/utils.ts.
